| 项 | 内容 |
|----|------|
| **方案版本** | v1.0.4 部署方案 |

## 一、部署背景

### 为什么升级

v1.0.0 → v1.0.4 主要变更：

1. **省级外综服认定能力**（P3 核心）
   - 新增 6 大服务模块：服务企业档案 / 物流 / 退税 / 结算 / 信保 / 融资
   - 新增认定驾驶舱 + Excel 导出
   - 业务数据链路打通（合同/订单/报关单互联）

2. **安全加固**（P0+P1）
   - SQL 注入、XSS、越权漏洞修复
   - 密码复杂度强制（8-32 位 + 大小写+数字+特殊字符）
   - 登录失败 5 次锁定 30 分钟
   - 全业务操作日志记录（AOP）

3. **运行时升级**
   - JDK 8 → JDK 17（jar 用 17 编译，必须升）
   - 数据库新增 7 张表 + 3 张表加字段

## 二、变更影响评估

| 维度 | 变更 | 影响 | 风险 |
|------|------|------|------|
| 后端 jar | 1.0.0 → 1.0.4 | 必须停服替换 | 中 |
| JDK 版本 | 8 → 17 | 需新装 JDK 17 | 中 |
| 数据库 | +7 表 +3 字段 | DDL 不影响旧数据 | 低 |
| 前端 dist | 全量替换 | 用户需 Ctrl+F5 | 低 |
| application-prod.yml | 不变 | 保留服务器原配置 | 无 |
| admin 密码 | 不动 | 保留旧密码 | 无 |
| 已有业务数据 | 不动 | 完整保留 | 无 |

## 四、部署执行步骤（按时序）

> 总时长预算 15 分钟，**实际经验**：8-10 分钟

### T+0：准备阶段（2 分钟）

```cmd
:: 1. 远程登录 192.168.3.65
:: 2. 进入部署目录
cd D:\deploy\autoWorkPlatform

:: 3. 上传新版部署包到临时目录 D:\deploy\new-1.0.4\
```

### T+2：备份阶段（3 分钟，**必做**）

```cmd
:: 1. 备份当前 jar
copy trade-platform-1.0.0.jar trade-platform-1.0.0.jar.bak

:: 2. 备份 start.bat
copy start.bat start.bat.bak

:: 3. 备份前端 dist
:: 假设 nginx 在 D:\phpstudy_pro\Extensions\Nginx1.25.3
cd D:\phpstudy_pro\Extensions\Nginx1.25.3\html
ren dist dist.bak
cd D:\deploy\autoWorkPlatform

:: 4. 备份数据库
"D:\phpstudy_pro\Extensions\MySQL8.0.12\bin\mysqldump.exe" -uroot -p<密码> trade_platform > D:\backup\trade_before_1.0.4.sql
:: 验证备份大小 > 1MB
dir D:\backup\trade_before_1.0.4.sql
```

✅ **检查点**：4 个备份文件全部存在，数据库备份 > 1MB

### T+5：停服阶段（30 秒）

```cmd
cd D:\deploy\autoWorkPlatform
stop.bat

:: 确认 8081 已释放
netstat -ano | findstr ":8081" | findstr LISTENING
:: 应无输出
```

✅ **检查点**：8081 端口已无监听

### T+6：数据库迁移（1 分钟）

```cmd
"D:\phpstudy_pro\Extensions\MySQL8.0.12\bin\mysql.exe" -uroot -p<密码> trade_platform < D:\deploy\new-1.0.4\migration-1.0.4.sql
:: 输出无 ERROR 即成功
```

✅ **检查点**：

```sql
-- 进 mysql 验证
SHOW TABLES LIKE 'biz_service_enterprise';   -- 应有结果
SHOW TABLES LIKE 'sys_operation_log';        -- 应有结果
DESC biz_contract;                            -- 应能看到 enterprise_id, contract_type
```

### T+7：替换文件（1 分钟）

```cmd

:: 1. 删除旧 jar

:: 2. 复制新 jar

:: 3. 替换 start.bat（新版用 JDK 17 绝对路径）

:: 4. 替换前端 dist
xcopy /E /I D:\deploy\new-1.0.4\dist D:\phpstudy_pro\Extensions\Nginx1.25.3\html\dist
```

✅ **检查点**：

```cmd

dir D:\phpstudy_pro\Extensions\Nginx1.25.3\html\dist\index.html
```

### T+8：启动后端（1 分钟）


预期输出：

## 五、数据库迁移脚本（migration-1.0.4.sql）

> 完整 SQL 见 [升级部署指南-v1.0.4.md](升级部署指南-v1.0.4.md) 第 3 步，或直接从 [sql/init.sql](sql/init.sql) 第 529 行往后复制 CREATE TABLE + ALTER TABLE 部分。

变更摘要：

| # | 操作 | 表/字段 |
|---|------|---------|
| 1 | CREATE | `biz_service_enterprise` 服务企业档案 |
| 2 | CREATE | `biz_logistics` 物流 |
| 3 | CREATE | `biz_tax_refund` 退税 |
| 4 | CREATE | `biz_settlement` 结算 |
| 5 | CREATE | `biz_insurance` 信保 |
| 6 | CREATE | `biz_financing` 融资 |
| 7 | CREATE | `sys_operation_log` 操作日志 |
| 8 | ALTER | `biz_contract` 加 enterprise_id, contract_type |
| 9 | ALTER | `biz_order` 加 enterprise_id |
| 10 | ALTER | `biz_customs_declaration` 加 contract_id, order_id, enterprise_id |

**风险评估**：低
- 只新增表/字段，不修改/删除现有结构
- 旧业务数据完全不受影响
- 失败的话单步可重试（CREATE TABLE IF NOT EXISTS 幂等；ALTER 需手工跳过已加字段）

## 六、回滚预案（5 分钟内执行）

**触发条件**（任一发生即回滚）：
- 后端启动后 5 分钟内反复重启
- 登录接口持续 500 错误
- 关键业务功能（合同/订单/报关）报错
- 用户上报严重影响业务的故障

**回滚步骤**：

```cmd
cd D:\deploy\autoWorkPlatform

:: 1. 停新版
stop.bat

:: 2. 恢复 jar
del trade-platform-1.0.4.jar
ren trade-platform-1.0.0.jar.bak trade-platform-1.0.0.jar
```

## 七、故障处理预案

### 故障 1：后端启动失败 `UnsupportedClassVersionError`

**原因**：start.bat 仍指向 JDK 8

**处理**：
```cmd
:: 1. 看 start.bat 第 8 行 JAVA17 变量是否指向 JDK 17
type start.bat | findstr JAVA17

:: 2. 验证 JDK 17 路径
"C:\Program Files\Zulu\zulu-17\bin\java.exe" -version

:: 3. 修正 start.bat 后重启
stop.bat
start.bat
```

### 故障 2：登录提示"系统繁忙"

**原因**：数据库表缺失（迁移没跑或部分失败）

**处理**：
```cmd
:: 查日志找具体错误
type logs\app.log | findstr "doesn't exist"

:: 看缺哪张表，单独建
:: 如缺 sys_operation_log，重跑迁移 SQL
```

### 故障 3：前端访问 502

**原因**：后端没起 / 8081 端口未监听

**处理**：
```cmd
netstat -ano | findstr ":8081"
:: 没监听就重启
stop.bat
start.bat

:: 仍失败看日志末尾 50 行
powershell -c "Get-Content logs\app.log -Tail 50"
```

### 故障 4：前端菜单缺新模块

**原因**：浏览器缓存 / dist 没替换

**处理**：
1. 用户浏览器 Ctrl+F5 强刷
2. 检查 nginx html\dist\index.html 文件时间是否是今天
3. 看 index.html 内引用的 hash 文件名是否变化

### 故障 5：登录提示"登录失败次数过多"

**原因**：升级前测试时密码错误累积触发锁定

**处理**：
- 等 30 分钟自动解锁
- 或重启后端清空内存锁定状态：`stop.bat && start.bat`
